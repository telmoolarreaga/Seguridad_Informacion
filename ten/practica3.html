Práctica 03

Vulnerabilidades XSS

Reflected XSS Stored XSS innerHTML localStorage
01

Reflected XSS (vulnerable)

¿Qué es?

En un ataque Reflected XSS, el payload malicioso se inyecta directamente en la URL o en un campo de entrada y se refleja de inmediato en la respuesta del servidor. El script se ejecuta en el navegador de la víctima cuando ésta visita el enlace manipulado.

// demo_reflected.js — VULNERABLE

Lo que escribas aquí se inserta usando innerHTML. Prueba el payload: <img src=x onerror="alert('XSS')">

$ resultado:
Código vulnerable
// VULNERABILIDAD
function updateReflected() {
  const input = getElementById('reflectedInput').value;
  result.innerHTML = input; // ← peligroso
}
¿Por qué es peligroso?

Usar innerHTML permite que el navegador interprete y ejecute HTML y JavaScript incrustado en la cadena. Si el contenido proviene del usuario sin sanitizar, un atacante puede inyectar scripts arbitrarios.

02

Stored XSS (vulnerable con localStorage)

¿Qué es?

En un ataque Stored XSS, el payload malicioso se almacena en el servidor (o en este caso, en localStorage) y se ejecuta cada vez que la página lo recupera y lo renderiza con innerHTML. Es más peligroso que el Reflected porque no requiere que la víctima haga clic en un enlace especial.

// demo_stored.js — VULNERABLE

Añade una nota. Se guarda en localStorage y luego se renderiza con innerHTML. Prueba: <script>alert('Stored XSS')</script>

$ notas_guardadas:
    Código vulnerable
    // VULNERABILIDAD
    function renderNotes() {
      const note = localStorage.getItem('userNote');
      noteList.innerHTML = `<li>${note}</li>`;
      // ← el payload persiste y se ejecuta
    }
    ¿Por qué es más peligroso?

    A diferencia del Reflected, el payload se persiste. Cada vez que la página cargue y llame a renderNotes(), el script almacenado se ejecutará automáticamente, afectando a cualquier usuario que visite la página.

    FIX

    Cómo Arreglarlo

    Solución principal

    Sustituir innerHTML por textContent. Con esta propiedad, el navegador trata el valor como texto plano, nunca como HTML o JavaScript, por lo que cualquier payload se mostrará literalmente en pantalla en lugar de ejecutarse.

    Reflected XSS — versión segura
    // SEGURO
    function updateReflected() {
      const input = getElementById('reflectedInput').value;
      result.textContent = input; // ← seguro
    }
    Stored XSS — versión segura
    // SEGURO
    function renderNotes() {
      const note = localStorage.getItem('userNote');
      const li = document.createElement('li');
      li.textContent = note; // ← seguro
      noteList.appendChild(li);
    }

    Tipos de XSS

    Reflected XSS

    El payload viaja en la petición (URL, formulario) y se devuelve en la respuesta inmediata. No persiste. Se distribuye mediante enlaces maliciosos enviados a la víctima.

    Stored XSS

    El payload se almacena en la base de datos, localStorage o cualquier otro medio persistente. Se ejecuta en cada visita. Mucho más peligroso por su alcance masivo.

    DOM-Based XSS

    El ataque ocurre íntegramente en el cliente. El payload nunca llega al servidor; se inyecta manipulando el DOM directamente a través de fuentes como location.hash, document.URL, etc.

    Payloads comunes
    • <script>alert('XSS')</script>
    • <img src=x onerror=alert(1)>
    • <svg onload=alert(1)>
    • javascript:alert(1) en href
    VA

    Valoración

    Utilidad

    Utilidad muy alta. XSS es una de las vulnerabilidades web más comunes (OWASP Top 10). Entender cómo funciona es fundamental para cualquier desarrollador o analista de seguridad.

    Dificultad

    Dificultad baja-media. Los conceptos son sencillos de entender, aunque existen variantes avanzadas (bypass de filtros, CSP evasion) que elevan la complejidad.

    ¿Qué hemos aprendido?

    La diferencia entre Reflected y Stored XSS, por qué innerHTML es peligroso con input de usuario, y cómo una simple sustitución por textContent mitiga completamente el riesgo en estos ejemplos.
